Naming Conventions (Reference)
================================

Used for: A single place to record naming rules for tables, columns, route names, and categorical values.
Information inside: The conventions chosen for this project so schema + code stay consistent.

1. General style
-----------------
- Use `snake_case` everywhere in identifiers (tables, columns, file names, and Python symbols).
- Be consistent about singular vs plural naming and stick to it everywhere.

2. Table names
---------------
Chosen convention: plural table names using `snake_case`.
- `users`
- `exercise_types`
- `workout_logs`
- `daily_metrics`
- `nutrition_logs`
- `goals`
- `daily_checkins`
- `progress_snapshots`
- `achievements`

3. Primary keys / foreign keys
--------------------------------
- Primary key column name: `id` (integer).
- Foreign key column naming: `<referenced_table_singular>_id`
  - Examples:
    - `users.id` referenced by `user_id` on user-owned tables
    - `goals.id` referenced by `goal_id` when applicable
- FK column naming must use the *singular* form of the referenced table name (even though table names are plural).
- Every user-owned record must include `user_id` and all CRUD operations must be scoped to the logged-in user.
- Index foreign keys: add an index for each FK column (e.g., `INDEX (user_id)`, `INDEX (goal_id)`) to speed up joins and WHERE lookups.

4. Column naming
------------------
- Use `snake_case` for column names.
- Use `created_at` and `updated_at` for timestamp columns (when needed).
- UTC timestamp rule: store all timestamp columns in UTC (e.g., `created_at`, `updated_at`, `deleted_at`).
- Soft delete (deleted_at): include a nullable `deleted_at` timestamp for user-owned tables that can be deleted; treat rows with `deleted_at IS NULL` as “active” and filter out soft-deleted records by default.
- Use boolean columns with `is_` prefix (e.g., `is_active`).
- Use `*_id` suffix for identifier columns.

5. Categorical / enum values
-------------------------------
- Store categorical values as lowercase strings with underscores (no spaces).
- Enum validation note: enforce allowed values in the DB where possible (e.g., `CHECK` constraints) and also validate at the app/service layer so invalid values are rejected before write.
- Examples:
  - `energy_level`: `low`, `medium`, `high`
  - `goal_status`: `active`, `completed`, `paused`
  - `meal_type`: `breakfast`, `lunch`, `dinner`, `snack`

6. API route naming
---------------------
- Use plural nouns for route paths where it reads cleanly:
  - `/goals`, `/metrics`, `/workouts`, `/nutrition`, `/checkins`, `/achievements`
- Use consistent route handler names in code (e.g., `get_goals`, `create_goal`, etc.).

7. Template naming
--------------------
- Template filenames match plural entity naming:
  - `goals.html`, `metrics.html`, `workouts.html`, `nutrition.html`, `checkins.html`, `achievements.html`
- Reuse `base.html` as the shared layout.

8. Stored procedure / view naming (if used)
---------------------------------------------
- Views: `vw_<entity>_<purpose>`
  - Example: `vw_dashboard_summaries`
- Procedures: `sp_<verb>_<entity>`
  - Example: `sp_create_daily_metric`

9. Source-of-truth rule
-------------------------
- `naming_struture.txt` is the source of truth.
- When schema and code conflict, update one side to match this file.
